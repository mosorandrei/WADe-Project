Technical Report: Botanical Gardens Application Development

Introduction

This technical report outlines key considerations and external data sources for the development of a web application focused on botanical gardens, as per the provided OpenAPI specification.

Initial Internal Data Structures/Models

  • User
    1. First Name
    2. Last Name
    3. Email
    4. Password
  • Botanical Garden
    1. Garden ID
    2. Name
    3. Attractions (List of Attractions)
  • Attraction
    1. Attraction ID
    2. Attraction Photo
    3. Name
    4. Type (e.g., Flower, Tree)
    5. Description
    6. Comments (List of Comments)
  • Tour
    1. Tour ID
    2. Name
    3. Description
    4. Attractions (List of Attractions)
    5. Comments (List of Comments)
  • Comment
    1. User
    2. Comment type (e.g., text, image)
    3. Comment content

External Data Sources to look into

  1. SPARQL Endpoint: Botanical information, taxonomy data, etc.
    • DBpedia
    • Wikidata
    • Bio2RDF
  2. Mapping Service: Garden maps (e.g., Google Maps API)
  3. Weather API: Real-time weather information
  4. Social Media APIs: Integration with platforms like Instagram, Twitter
  5. Educational Content APIs: Additional information about plant species and botanical sciences
  6. Events Calendar API: Upcoming activities, workshops, exhibitions
  7. Botanical Research Databases: Specific to botanical research and taxonomy

Linked Data Principles in the Botanical Gardens Application

The development of the Botanical Gardens Application adheres to the Linked Data principles, fostering a more interconnected and discoverable web of botanical information. Here's how the application aligns with each principle:

  1. Use URIs as Names for Things:

    The application assigns unique URIs to each botanical garden, attraction, tour, and comment, ensuring global identification and persistence.

  2. Use HTTP URIs:

    All URIs in the application use the HTTP protocol, allowing them to be dereferenced to retrieve information about the identified resource directly.

  3. Provide Useful Information Using Standardized Formats (e.g., RDF):

    The application provides information in RDF, a standardized and machine-readable format, enhancing the flexibility and extensibility of data representation.

  4. Include Links to Other URIs:

    Within the data, the application incorporates links to related URIs, creating a network of interconnected botanical information. Users and applications can navigate seamlessly between resources.

  5. Link to Other Data Sources:

    The application extends beyond its own dataset by linking to external data sources, contributing to the creation of a decentralized and distributed web of interlinked botanical data.

By aligning with these Linked Data principles, the Botanical Gardens Application promotes a more open, interconnected, and accessible platform for botanical information, enriching the experience for users and encouraging collaboration with external data providers.

Conclusion

The development of the Botanical Gardens Application involves a comprehensive approach, incorporating internal data structures/models, and integrating various external data sources to enhance user experience and functionality.
